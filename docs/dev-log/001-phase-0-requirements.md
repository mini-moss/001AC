## 为什么每一条这么写

### 1. 硬件平台：为什么 ARM + RISC-V？

- **单一架构**无法逼出可移植的 HAL 层（你会在 cortex-m 的特殊寄存器里写太多糖）。
- ARM Cortex-M4 是**机器人主控的事实标准**（STM32 全家桶），你做的任何东西都能跑。
- RISC-V 让你**被迫面对第二个 ISA**，这是写 RTOS 真正有学习价值的部分。
- 只选这两个 → 范围可控；想再加 ESP32 / AVR 是 Phase 10 之后的事。

### 2. 实时性指标：为什么用「表格 + 测量方法」？

RTOS 圈最常见的翻车是「我们假设中断延迟 1 μs，结果实测 50 μs」。把**测量方法**写进需求，等于**写了一个验收测试的雏形**。

- RAM < 32 KB / ROM < 128 KB 不是随手写的：STM32F407 入门级有 192 KB RAM、1 MB Flash，留 1/6 空间给应用是合理目标。
- Tick 周期 1 ms 是**默认**，100 μs 是**高精度档**（要权衡中断开销），用配置宏切。

### 3. 调度策略：为什么 EDF 是「可选」？

- 固定优先级 + 时间片轮转能覆盖 90% 机器人场景。
- EDF 理论上能跑满 100% 利用率，但**实现复杂度高、可分析性差**，机器人项目几乎用不上。
- 把 EDF 放在「插件」位置 → 你以后想研究它可以加，**不会污染主调度器**。

### 4. API 风格：为什么要强调「是否可在 ISR 中调用」？

- ISR 中**不能阻塞**（不能 `sem_take` 等待），所以 ISR-safe API 的实现必须**不关调度**或**用 lock-free 数据结构**。
- 提前标注 → 写实现时不用每次重新设计，写文档时一眼就能给用户答案。
- FreeRTOS 用 `*FromISR` 后缀区分；我们这里用**文档标注**而不是函数后缀，更紧凑。

### 5. 许可协议：为什么挑 Apache 2.0 而不是 MIT？

- MIT 允许一切，**不含专利授权**。
- Apache 2.0 **明确授予专利使用权**，对工业机器人（专利环境复杂）友好。
- 公司内部用 → Apache 比 GPL 友好；个人/学术用 → MIT 也可以。
- 「不同模块不同协议」是行业惯例（lwIP / LVGL / micro-ROS 都各管各的）。

### 6. 「不做什么」是这文档里**最值钱的部分**

写过 RTOS / 嵌入式的人都知道：scope creep 是头号死因。
把「不做 MMU」「不做 SMP」「不做 POSIX 严格兼容」**写下来**，未来有人说「加个虚拟内存吧」你可以直接指回这里。

## 哪些地方你可能想改

我**替你拍板了**几件事，你过一眼，**不同意就直接说**：

1. **任务数上限 32**：太小（FreeRTOS 默认无上限）、太大（占 RAM）。改 16/64 都行。
2. **FatFS 协议问题**：我标了「需评估」。如果不做文件系统，可以整个删掉第 5 节那条 FatFS 警告。
3. **不做 SMP**：这是 v1.0 的硬限制。如果你的机器人主控是双核 M7，告诉我，我们改 v1.0 范围。
4. **Cortex-M4 起步**：如果你只有 M0+ 的板子（比如 STM32G0），告诉我，调成 M0+。
