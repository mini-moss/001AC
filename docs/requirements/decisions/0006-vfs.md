# ADR-0006 — VFS 抽象（vfat 后端 + 字符设备）

> **Status**：Accepted（v0.3.1 新建）
> **Date**：2026-06-07
> **Supersedes**：无（v0.3.1 新建）
> **Deciders**：junbo.dai
> **Refs**：[requirements §4.2](../requirements.md#42-用户态-syscall用户进程跨边界调用) · [requirements §9](../requirements.md#9-v10-sub-scope-详细清单) · [ADR-0003](0003-license.md) · [ADR-0005](0005-user-kernel-boundary.md) · [dev-log/005](../../dev-log/005-unified-hybrid-kernel-v1-subscope.md)

## Context

v0.3.1 sub-scope 要求：

- syscall `sys_open / sys_read / sys_write / sys_close`（[§4.2](../requirements.md#42-用户态-syscall用户进程跨边界调用)）
- VFS：vfat + 字符设备（[§9 sub-scope](../requirements.md#9-v10-sub-scope-详细清单)）

需要回答：

1. **VFS 是分层抽象还是直接调用**？
2. **后端有哪些**？（vfat / char device / 其他？）
3. **vfat 用哪个库**？（FatFs / Petit FatFs / FatFs IoT / 自写）
4. **FD 表是 per-process 还是 per-thread**？
5. **路径解析**：绝对路径？支持 cwd？支持 mount？
6. **字符设备命名**：`/dev/uart0` 风格还是其他？
7. **错误码**：Linux 风格 `-EINVAL` 等

---

## Decision

### VFS 架构：分层 + file_ops vtable（Linux 风格）

```c
struct file_ops {
    ssize_t (*read)(struct file *f, void *buf, size_t n, off_t *pos);
    ssize_t (*write)(struct file *f, const void *buf, size_t n, off_t *pos);
    int     (*open)(struct file *f, int flags);
    int     (*close)(struct file *f);
};

struct file {
    const struct file_ops *ops;
    void   *priv;       // 后端私有数据（vfat inode / char dev handle）
    int     flags;
    off_t   pos;
    int     refcnt;
};
```

### v1.0 后端

| 后端 | 路径前缀 | 用途 | 实现 |
|---|---|---|---|
| vfat | `/vfat/` | 块设备文件 | FatFs IoT 库（Apache 2.0 兼容） |
| char | `/dev/` | UART / console | 内核内嵌驱动表 |

### FD 表

- **per-process FD 表**，固定 32 槽位
- FD 0–2 默认指向 `/dev/console`（stdin / stdout / stderr）
- `sys_open` 返回最小空闲 FD（Linux 习惯）
- 进程退出时 kernel 自动 close 所有 open FD

### 路径解析

- **仅绝对路径**：必须以 `/` 开头（v1.0 无 cwd 概念）
- **无 mount points**：路径前缀固定（`/vfat/` → vfat 后端；`/dev/` → char 后端）
- 其他前缀 → `-ENOENT`

### 字符设备表

```c
struct char_device {
    const char *name;          // "uart0", "console"
    const struct file_ops *ops;
};
// v1.0 内嵌：console, uart0, uart1（按板级 BSP 决定）
```

### 错误码（Linux 兼容）

| 错误 | 值 | 场景 |
|---|---|---|
| `-EBADF` | -9 | FD 无效 |
| `-ENOENT` | -2 | 文件不存在 |
| `-EACCES` | -13 | 权限不足（v1.0 仅 vfat 只读时） |
| `-EFAULT` | -14 | 用户指针错（per ADR-0005） |
| `-EMFILE` | -24 | FD 表满 |
| `-ENOSYS` | -38 | 不支持的操作（如 `sys_open` 不支持目录） |

## Consequences

### 积极

- ✅ **统一接口**：`sys_read / sys_write` 不关心后面是 vfat 还是 char，应用代码可移植
- ✅ **可扩展**：v2.0 加 ext4 / procfs / devfs 只需新写 file_ops 实现
- ✅ **Linux ABI 兼容**：FD + 错误码 + 路径形式跟 Linux 一致，user-space 工具可移植
- ✅ **FatFs IoT 协议兼容**：Apache 2.0 兼容 BSD-like 协议，无 [ADR-0003](0003-license.md) 阻塞
- ✅ **per-process FD 表 + 进程退出自动 close**：避免 FD 泄漏污染系统

### 消极

- ❌ **无 mount points**：路径前缀写死，加新后端要改路径解析逻辑
- ❌ **无 cwd**：用户必须传绝对路径，shell 实现要自己维护"逻辑 cwd"
- ❌ **vfat 用第三方库**：FatFs IoT 的 bug / 行为差异要查上游
- ❌ **32 FD 上限**：sub-scope 内够，但 micro-ROS 类型大消息可能有压力
- ❌ **char 设备表静态**：v1.0 不支持运行时 register char dev（v1.1+ 加 `register_char_dev()`）

### 中性

- VFS 头文件接口跟 Linux 不严格一致（如 `struct inode` 不暴露给驱动），方便 v1.0 简化
- 大部分 Linux 文件系统语义（dentry、namespace）**v1.0 不实现**，文档化为"未来扩展点"

---

## Alternatives Considered

### 备选 A：无 VFS，直接 syscall 路由

- ✅ 极简，代码量少
- ❌ 每加一个后端，`sys_read` 等都要改分支
- ❌ user-space 代码无法跨后端移植
- **结论**：pass

### 备选 B：完整 Linux VFS（含 inode / dentry / mount / namespace）

- ✅ 真正可移植
- ❌ ~5000+ 行代码，超 v1.0 sub-scope
- **结论**：v2.0+ 评估

### 备选 C：FatFs（旧版，非 IoT）

- ✅ 成熟，文档全
- ❌ **协议非 OSI**（per [ADR-0003](0003-license.md)）
- **结论**：pass（已在 ADR-0003 拒绝）

### 备选 D：自写 vfat

- ✅ 完全控制，无外部依赖
- ❌ 工作量 ~2 月，超 v1.0 sub-scope；vfat 格式细节多
- **结论**：pass

### 备选 E：用 littlefs 代替 vfat

- ✅ 嵌入式优化，磨损均衡
- ❌ littlefs 是为 raw flash 设计，不读 PC 上的 vfat 镜像；跟 SD 卡 / U 盘场景不匹配
- **结论**：v1.x+ 评估为 NOR flash 备选

## 关键设计细节（影响后续 Phase）

### 块设备接口（vfat 后端依赖）

```c
struct block_device {
    int  (*read_block)(uint32_t lba, void *buf);     // 一次 1 block (512 B)
    int  (*write_block)(uint32_t lba, const void *buf);
    uint32_t block_count;
};
```

- vfat 后端调 block_device 接口；底层由 SD/EMMC 驱动实现（Phase 6）
- QEMU 测试用 `-drive file=test.img`（virtio-blk 模拟）

### 字符设备注册（v1.0 静态表）

```c
extern const struct char_device char_devices[];
extern const int char_devices_count;

// 板级 BSP 在 boards/<board>/devices.c 填充
const struct char_device char_devices[] = {
    { "console", &console_ops },
    { "uart0",   &uart0_ops   },
};
```

### sys_open 路径解析（v1.0 实现）

```c
int sys_open(const char __user *upath, int flags) {
    char path[256];
    if (copy_from_user(path, upath, 256) < 0) return -EFAULT;

    if (strncmp(path, "/vfat/", 6) == 0)
        return vfat_open(path + 6, flags);
    if (strncmp(path, "/dev/", 5) == 0)
        return chardev_open(path + 5, flags);
    return -ENOENT;
}
```

### Phase 实现路径

- **Phase 5**：file_t / file_ops 抽象 + FD 表 + console 字符设备（hello world 走 stdout）
- **Phase 6**：SD/EMMC block 驱动 + FatFs IoT 集成 + uart 字符设备
- **Phase 8**：vfat 镜像在 QEMU virtio-blk 上读写测试
- **Phase 9**：FD 泄漏 / 路径越界 fuzz 测试

---

## Validation

- [ ] **Phase 5 末**：`sys_write(1, "hello\n", 6)` 输出到 console（默认 FD 1 = /dev/console）
- [ ] **Phase 6 末**：QEMU 挂载 vfat 镜像，`sys_open("/vfat/test.txt", O_RDONLY)` 读到正确内容
- [ ] **Phase 6 末**：`sys_open("/dev/uart0", O_RDWR)` + `sys_write` 输出到 UART
- [ ] **Phase 6 末**：超过 32 FD → `sys_open` 返回 `-EMFILE`
- [ ] **Phase 6 末**：访问 `/proc/foo` → `-ENOENT`（v1.0 无 procfs）
- [ ] **Phase 9 末**：进程退出后 kernel 检查 FD 表，所有 open FD 全部 close
- [ ] **Phase 9 末**：路径越界 fuzz：随机长度 / 字符的 path，应永远不崩内核

任一项不达标 → 评估降级（如把 vfat 推迟到 v1.1，v1.0 仅 char dev）。

---

## Notes

- 本 ADR 跟 [ADR-0005 syscall ABI](0005-user-kernel-boundary.md) 联动：`sys_open/read/write/close` 路径都从 syscall 进
- FatFs IoT 库的选型最终确认要在 Phase 6 开工前做（评估协议、维护活跃度、ARM/RV64 支持）
- v2.0+ 评估方向：mount points / namespace / ext4 / devfs / procfs / 网络 socket-fs
- 块设备 buffer cache **v1.0 不做**——每次读直接走块设备；v1.1+ 评估 LRU cache
